# 問題点の多角的分析と対応状況

> 分析日: 2026-07-08 ／ 対象: 全コード（domain / store / components / lib / api / CI）
> UX_AUDIT.md で対応済みの A-1（食事の逆インセンティブ）・A-2（ストリーク基準）は除く。
> 本ドキュメントは「残っている問題」のトラッカー。解決したら ✅ を付ける。

## A. データの信頼性・消失リスク（最優先）

- [x] **A-1 トレ記録の取り消しが無い** ✅ 対応済み
  重量タイプミス（40→400kg）が巨大EXP・自己ベスト・ボスダメージとして永久確定し、
  修正手段が無かった。→ 記録直後（当日・最新の1件）の取り消しを実装。
  EXP/レベル/ステータス/ゴールド/ボス/部位ボリューム/自己ベスト/ストリーク/HP/
  消費アイテムをすべて巻き戻す。
- [x] **A-2 localStorage のみの永続化・バックアップ手段なし** ✅ 一次対応済み
  機種変・キャッシュクリアで育てたアバターが全消失する。
  → 設定画面にバックアップの書き出し（JSONダウンロード）と読み込みを追加。
  根本対応（Supabase 匿名認証＋自動同期）は Phase 2 の最優先のまま。
- [ ] **A-3 persist にバージョン移行（migrate）が無い**
  スキーマ変更時に既存ユーザーのデータが壊れても検知できない。
  現状は `??` フォールバックで凌いでいるが、`version` + `migrate` の導入を推奨。

## B. セキュリティ

- [ ] **B-1 Gemini API キーがクライアントバンドルに露出**（`VITE_GEMINI_API_KEY`）
  誰でも DevTools からキーを抽出して悪用できる＝課金リスク。
  Yahoo 同様にサーバー関数経由へ移すべきだが、現デプロイ先が GitHub Pages
  （サーバー無し）のため **ホスティングの判断が必要**（B-3 と同根）。
  暫定策: Google AI Studio 側でキーにリファラ制限を掛ける。
- [x] **B-2 コミュニティDBの異常値がそのまま流れ込む** ✅ 一次対応済み
  誰でも任意バーコードを upsert でき、負の値や protein 99999g などの異常値も
  全ユーザーの記録・コンディション計算に直行していた。
  → クライアント側で参照時・登録時の両方に値のクランプ（0〜上限・非数排除）を追加。
  根本対応は Supabase 側の RLS / CHECK 制約（要ダッシュボード作業）。
- [ ] **B-3 デプロイ先が2つ併存している**
  GitHub Pages（`.github/workflows/deploy.yml`）と Vercel（PRプレビューも
  デプロイされている）の両方に配信されている。Pages 側にはサーバーが無いため
  `/api/yahoo-jan` が常に 404 → Yahoo 商品名検索が機能しない（Open Food Facts
  フォールバックのみ生存）。Vercel 側では機能する。ユーザーがどちらのURLを
  使うかで挙動が変わるのは危険なので、Vercel へ一本化して Pages ワークフローを
  廃止するのが自然（B-1 のAPIキー秘匿もVercel Functionsで解決できる）。
  **オーナー判断待ち**。

## C. ゲームデザイン／UX（UX_AUDIT 未対応分）

- [x] **C-1 復帰の瞬間を罰する演出**（UX_AUDIT A-3） ✅ 対応済み
  数日ぶりに開くと開幕「😈サボった！-HP」のフルスクリーンが最初の体験で、
  再離脱の最大要因になりうる。→ 「おかえり！」側にフレーミングを反転。
  ダメージの事実は残しつつ、「今日トレすれば回復する」を主役に。
- [x] **C-2 睡眠ポップアップが毎起動割り込み** ✅ 対応済み
  「あとで」がセッション内のみ有効で、再起動のたびに全画面再表示だった。
  → 「あとで」を永続化し 1日1回までに（記録は従来どおりホームからいつでも可能）。
- [ ] **C-3 クエストの初心者ハードル**
  「2部位以上」が毎日並んで未達続き＝学習性無力感。タンパク質×1.6g/日も
  常時未達になりやすい。段階制（×1.2→×1.6）や曜日ローテを推奨。
  **難度設計の判断が要るためオーナーと相談**。
- [ ] **C-4 HP が実質飾り**
  下限1で死なず、減っても何の帰結も無い（脅しの演出だけがある状態）。
  「HPが低いと獲得EXPが下がる」等の軽い意味付け、または撤去を検討。

## D. コード品質・パフォーマンス

- [ ] **D-1 バンドル 666KB（gzip 187KB）が単一チャンク**
  zxing（バーコード）等の重量級が初回ロードに全部載る。
  `React.lazy` でスキャナーを遅延ロードすれば初回表示が軽くなる。
- [ ] **D-2 `logWorkout` が170行のモノリスで未テスト**
  ストリークシールド・ボス・自己ベスト更新など最重要ロジックがストアに集中し、
  テストは domain 層のみ。ロジックの domain 抽出＋テスト追加を推奨。
- [ ] **D-3 `selectToday` が毎回新オブジェクトを返す**
  購読コンポーネントがストア変更のたびに再レンダーされる（実害は小）。
- [ ] **D-4 エラーバウンダリ無し**
  描画エラー1つで白画面になる。データは localStorage にあるので復旧は可能だが、
  ユーザーには全損に見える。
- [ ] **D-5 `claimQuest` が報酬値を UI から信用して受け取る**
  ローカルアプリなので実害は小さいが、サーバー移植時に要修正。
