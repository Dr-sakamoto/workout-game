# 問題点の多角的分析と対応状況

> 分析日: 2026-07-08 ／ 対象: 全コード（domain / store / components / lib / api / CI）
> UX_AUDIT.md で対応済みの A-1（食事の逆インセンティブ）・A-2（ストリーク基準）は除く。
> 本ドキュメントは「残っている問題」のトラッカー。解決したら ✅ を付ける。

## A. データの信頼性・消失リスク（最優先）

- [x] **A-1 トレ記録の取り消しが無い** ✅ 対応済み
  重量タイプミス（40→400kg）が巨大EXP・自己ベスト・ボスダメージとして永久確定し、
  修正手段が無かった。→ 記録直後（当日・最新の1件）の取り消しを実装。
  EXP/レベル/ステータス/ゴールド/ボス/部位ボリューム/自己ベスト/ストリーク/HP/
  消費アイテムをすべて巻き戻す。
- [x] **A-2 localStorage のみの永続化・バックアップ手段なし** ✅ 一次対応済み / 根本対応P1・P2実装済み
  機種変・キャッシュクリアで育てたアバターが全消失する。
  → 設定画面にバックアップの書き出し（JSONダウンロード）と読み込みを追加（一次対応）。
  根本対応（Supabase 匿名認証＋メール連携＋同期）の方針を [SYNC_DESIGN.md](./SYNC_DESIGN.md)
  に確定し、**P1（スキーマ）・P2（同期コア: 起動時プル/デバウンスプッシュ/進捗ガード/
  設定トグル）を実装済み**。残るP3（メール連携UI）・P4（ログ単位マージ）は
  SYNC_DESIGN.mdに仕様確定済み。
  ⚠️ マイグレーション適用とSupabaseダッシュボードでの匿名サインイン有効化が必要
  （未実施でも実害なし＝ローカルのみで動き続けるだけ）。
- [x] **A-3 persist にバージョン移行（migrate）が無い** ✅ 対応済み
  スキーマ変更時に既存ユーザーのデータが壊れる/新フィールドが undefined になる
  リスクがあった。→ `version` + `migrate` を導入し、FRESH の初期値を土台に
  永続化値を上書き（ネストしたオブジェクトも欠落キーを補う）。

## B. セキュリティ

- [x] **B-1 Gemini API キーがクライアントバンドルに露出**（旧 `VITE_GEMINI_API_KEY`） ✅ 対応済み
  誰でも DevTools からキーを抽出して悪用できる＝課金リスクだった。
  → Yahoo 同様に Vercel サーバーレス関数（`api/gemini-nutrition.ts`）経由へ移し、
  キーを `GEMINI_API_KEY`（VITE_ 無し・サーバー環境変数）に変更。バンドルから消えた。
  ※Vercel 側の環境変数に `GEMINI_API_KEY` を設定すること（デプロイ時作業）。
- [x] **B-2 コミュニティDBの異常値がそのまま流れ込む** ✅ 対応済み（クライアント＋DB）
  誰でも任意バーコードを upsert でき、負の値や protein 99999g などの異常値も
  全ユーザーの記録・コンディション計算に直行していた。
  → クライアント側で参照時・登録時の両方に値のクランプ（0〜上限・非数排除）を追加。
  根本対応も実装済み: `community_foods` へ CHECK 制約（P1）＋書き込みを認証済み
  （匿名可）ユーザーに限定するRLS（P2、`supabase/migrations/20260710_0002_*.sql`）。
  ⚠️ マイグレーション適用が必要。
- [x] **B-3 デプロイ先が2つ併存していた** ✅ 対応済み
  GitHub Pages（`.github/workflows/deploy.yml`）と Vercel の両方に配信され、
  Pages 側では `/api/*` が常に 404、Vercel 側では動く＝URLで挙動が変わる状態だった。
  → GitHub Pages ワークフローを削除し、Vercel へ一本化。これで `api/yahoo-jan.ts`・
  `api/gemini-nutrition.ts` が本番で機能する。
  ※GitHub 側の Pages 設定（Settings → Pages）を無効化しておくと二重配信が完全に消える。

## C. ゲームデザイン／UX（UX_AUDIT 未対応分）

- [x] **C-1 復帰の瞬間を罰する演出**（UX_AUDIT A-3） ✅ 対応済み
  数日ぶりに開くと開幕「😈サボった！-HP」のフルスクリーンが最初の体験で、
  再離脱の最大要因になりうる。→ 「おかえり！」側にフレーミングを反転。
  ダメージの事実は残しつつ、「今日トレすれば回復する」を主役に。
- [x] **C-2 睡眠ポップアップが毎起動割り込み** ✅ 対応済み
  「あとで」がセッション内のみ有効で、再起動のたびに全画面再表示だった。
  → 「あとで」を永続化し 1日1回までに（記録は従来どおりホームからいつでも可能）。
- [x] **C-3 クエストの初心者ハードル** ✅ 対応済み
  「2部位以上」が毎日並んで未達続き＝学習性無力感、タンパク質×1.6g/日も
  常時未達になりやすかった。→ 記録するだけの低ハードル枠を2つ常設（トレ1回・
  食事1回）、「2部位以上」を「食事を1回記録する」へ置換、タンパク質目標を
  レベル連動の段階制（×1.2→×1.4→×1.6）に変更。
- [x] **C-4 HP が実質飾り** ✅ 対応済み
  下限1で死なず、減っても何の帰結も無かった（脅しの演出だけ）。
  → HPが0まで減るようにし、0になるとアバターの見た目が一時的に1段階なまる
  （`weakenedBuild`）。実データ（partVolumes）は減らさず、トレーニングで
  HPが戻れば見た目も元通り。「鍛えた肉体こそ報酬」（原則1）を使った可逆の復帰動機。

## D. コード品質・パフォーマンス

- [x] **D-1 バンドル 666KB（gzip 187KB）が単一チャンク** ✅ 対応済み
  zxing（バーコード）等の重量級が初回ロードに全部載っていた。
  → `React.lazy`+`Suspense` でスキャナーを遅延ロード。初回JSを
  187KB→73KB(gzip) に削減、スキャナ本体はオンデマンド化。
- [x] **D-2 `logWorkout` が170行のモノリスで未テスト** ✅ 対応済み
  ストリークシールド・ボス・自己ベスト更新など最重要ロジックがストアに集中し、
  テストは domain 層のみだった。→ `domain/workoutReducer.ts` に純関数
  `applyWorkoutLog` として抽出（EXP/ボス/ストリーク/自己ベスト/HP/取り消し
  スナップショットの計算を1関数に集約）。ストアは id/today の発行と
  set()呼び出しのみ。テスト16件追加（`workoutReducer.test.ts`）。
  副次効果: 将来のログ単位マージ（SYNC_DESIGN.md P4）が、このリデューサの
  上に素直に載せられるようになった。
- [ ] **D-3 `selectToday` が毎回新オブジェクトを返す**
  購読コンポーネントがストア変更のたびに再レンダーされる（実害は小）。
- [x] **D-4 エラーバウンダリ無し** ✅ 対応済み
  描画エラー1つで白画面になり、データは無事でも全損に見えた。
  → `ErrorBoundary` を導入し、「データは端末内に残っている」旨と
  再読み込み・バックアップ導線を提示する復旧画面を表示。
- [ ] **D-5 `claimQuest` が報酬値を UI から信用して受け取る**
  ローカルアプリなので実害は小さいが、サーバー移植時に要修正。
